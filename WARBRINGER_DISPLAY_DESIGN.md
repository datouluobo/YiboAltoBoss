# 战争使者与显示项配置设计

用途：沉淀当前关于“赞达拉战争使者监控方式”和“设置页显示项配置”的讨论结论，作为后续实施依据。

状态：设计持续更新中。目标模型与显示项配置已进入实施；主窗口交互方案以下方“更新附录”为最新结论。

---

## 1. 背景与问题

当前插件对目标的核心建模偏向“一个目标 = 一个 NPC ID”。这对常规世界 Boss 适用，但不适用于赞达拉战争使者。

战争使者的特殊点：
- 会在 5 张地图分别刷新。
- 一共有 3 个 NPC ID，对应 3 种颜色与坐骑掉落。
- 单张地图在某次刷新时，只会出现 3 种颜色中的 1 种。
- 同一个 NPC ID 可能同时或先后出现在不同地图，因此不能用单 `bossId` 唯一标识一个地图点位。
- 刷新间隔目前怀疑约为 30 分钟，但仍需继续观察验证。

现有实现下，如果直接把 `69769 / 69841 / 69842` 当作 3 个普通目标加入监控，会出现以下问题：
- 地图 1 杀掉 `ID1` 后，地图 2 再看到 `ID1`，会被错误视作同一个目标的延续。
- 位面记录、击杀记录、刷新样本会串到一起。
- 主窗口下半区无法准确表达“每张地图各自独立刷新”的状态。

---

## 2. 已确认的设计结论

### 2.1 战争使者改为内置硬编码

不再依赖“用户手工输入单个 NPC ID”来支持战争使者。

改为：
- 像世界 Boss 一样，作为插件内置目标硬编码。
- 但硬编码方式不能继续沿用“单 ID 目标”模型。
- 需要新增一类“地点型监控对象”。

### 2.2 战争使者按地图独立建模

战争使者不是共享同一个刷新池。

最终建模原则：
- 每张地图的战争使者是一个独立监控对象。
- 每个对象允许匹配 3 个颜色 ID 中的任意一个。
- 击杀记录、位面记录、刷新估计都按地图独立保存。

也就是说：
- 监控对象不是 `bossId`
- 监控对象是“地图刷新点”

### 2.3 主窗口保持现有上下两部分布局

主窗口继续保持当前结构：
- 上半部分：击杀矩阵
- 下半部分：位面 / 观察 / 刷新信息矩阵

不在主窗口中新增“分组显示”。

所有目标：
- 只在设置页里分组管理
- 在主窗口中继续作为一整份目标列表展示

### 2.4 设置页改为“组开关 + 单项开关”

设置页新增“显示项配置”区，按组展示目标。

已确认规则：
- 组开关控制整组是否启用。
- 单项开关控制组内每个目标是否显示。
- 当组开关关闭时，组内单项开关变灰且不可操作。
- 所有目标关闭显示时，停止该目标的观测与记录。
- 停止记录时，不删除配置文件里已有的历史记录。

### 2.5 显示关闭即停止记录

对任一目标，若它在设置中被关闭显示，则：
- 主窗口不显示该目标
- 观测逻辑不再继续记录该目标的新数据
- 击杀逻辑不再继续记录该目标的新数据
- 已有存档保留，不自动清除

这条规则适用于：
- 世界 Boss
- 战争使者
- 未来可能加入的其它内置特殊目标
- 仍保留的自定义目标

## 3. 推荐目标模型

## 3.1 目标分类

后续目标建议分成两类：

### A. 单 ID 目标

适用于常规世界 Boss。

示例：
```lua
{
  key = "nalak",
  type = "single_npc",
  id = 69099,
  name = "纳拉克",
  reset = "weekly",
  order = 10,
  group = "world_boss",
}
```

### B. 地点型目标

适用于战争使者这类“一个地图点允许多个 NPC ID”的目标。

示例：
```lua
{
  key = "warbringer_jade_forest",
  type = "location_rule",
  name = "战争使者·翡翠林",
  allowedNpcIds = { 69769, 69841, 69842 },
  zone = "翡翠林",
  reset = "manual",
  order = 210,
  group = "warbringer",
}
```

核心含义：
- `key` 是目标主键，后续记录建议都逐步以它为中心。
- `type` 区分匹配方式。
- `allowedNpcIds` 表示允许命中的 NPC 集合。
- `zone` 用于将命中的 NPC 归属到具体地图目标。
- `group` 用于设置页分组，不用于主窗口分组展示。

---

## 4. 战争使者内置目标清单

本轮讨论中，战争使者按 5 张地图内置。

建议条目：
- 战争使者·翡翠林
- 战争使者·昆莱山
- 战争使者·螳螂高原
- 战争使者·卡桑琅丛林
- 战争使者·恐惧废土

每个条目共享同一组允许的 NPC ID：
- `69769`
- `69841`
- `69842`

建议先按 `zone` 匹配。

后续如发现仅靠地图名不够稳定，再考虑升级为：
- `mapId`
- 或 `zone + 子区域`

当前优先级：
1. `zone`
2. 必要时补 `mapId`
3. 非必要不先引入更复杂的多条件匹配

---

## 5. 设置页设计

## 5.1 设计目标

设置页需要从“工具型配置入口”升级为“显示项管理入口”，但仍保持紧凑、清晰、可信赖的产品气质。

目标：
- 用户能快速决定主窗口展示哪些目标。
- 特殊目标不再依赖手工输入复杂规则。
- 关闭某项后，用户能明确知道该项会停止记录，但历史不删除。

## 5.2 分组结构

建议至少提供以下分组：
- 世界 Boss
- 战争使者
- 自定义目标

如未来新增其它内置特殊监控，可继续增加组，但不改主窗口结构。

## 5.3 交互规则

每个分组包含：
- 一个组开关
- 多个单项开关

规则如下：
- 组开关关闭：整组目标停止显示与记录。
- 组开关关闭时：组内所有单项开关变灰。
- 组开关开启：组内单项开关恢复可操作。
- 单项开关关闭：该目标单独停止显示与记录。
- 单项开关开启：该目标恢复显示与记录。

## 5.4 文案建议

设置页建议明确提示：
- “仅勾选的项目会显示在主窗口中”
- “关闭显示会停止后续记录，但不会删除已有历史数据”

这条提示要尽量靠近显示项配置区域，减少误解。

## 5.5 与现有自定义目标输入的关系

本轮已确认战争使者改为硬编码，因此不再依赖“输入 NPC ID”方式处理战争使者。

对现有自定义目标输入功能，建议：
- 短期保留
- 仍作为通用补充能力存在
- 但不承担战争使者这种复杂目标的配置职责

后续若继续演进，可再讨论是否把“自定义目标”也升级为更完整的规则配置。

---

## 6. 主窗口设计

## 6.1 总体原则

主窗口保持当前显示方式，不按分组显示。

也就是说：
- 设置页中有分组
- 主窗口中无分组标题
- 主窗口只显示“当前启用的目标列表”

## 6.2 上半区

上半区继续作为击杀矩阵使用。

世界 Boss：
- 行为不变，继续按现有周常击杀逻辑显示。

战争使者：
- 每张地图对应一行。
- 击杀状态记录的是“该地图对应战争使者目标是否被击杀/观测到击杀”。

## 6.3 下半区

下半区继续作为位面 / 观察 / 刷新信息矩阵。

战争使者在这里的意义更强：
- 每行表示某张地图的战争使者状态。
- 位面、观察时间、击杀计时、刷新估计都按该地图单独呈现。

## 6.4 排序建议

主窗口仍建议保持稳定排序，避免用户视觉记忆被打乱。

建议顺序：
- 现有世界 Boss 保持原顺序
- 战争使者作为内置目标追加在后
- 自定义目标继续排在最后

---

## 7. 记录与匹配规则

## 7.1 目标命中规则

### 单 ID 目标

命中条件：
- 观察或击杀到该 `id`

### 战争使者地点型目标

命中条件：
- 当前 NPC ID 属于 `allowedNpcIds`
- 当前地图 / 区域匹配该条目的 `zone`

满足两者时，才把这次观察或击杀归属到对应地图目标。

## 7.2 记录停止规则

目标被关闭显示后：
- 不再参与观察匹配
- 不再参与击杀记录
- 不再在主窗口和悬停窗口显示

但保留：
- 已有击杀记录
- 已有位面信息
- 已有刷新样本

## 7.3 历史数据保留规则

关闭显示不是删除。

因此：
- 不需要主动清理 SavedVariables 中已有记录
- 重新开启后，可继续看到此前累计的信息

是否需要在重新开启时立即恢复展示旧记录：
- 当前建议：恢复展示
- 这样更符合“关闭只是暂停记录”的用户预期

---

## 8. 与现有代码结构的对应关系

当前代码的一个关键限制是：默认目标缓存主要按 `boss.id` 建立查找。

现状表现：
- 默认列表是 `DEFAULT_BOSSES`
- 目标缓存是 `bossById[boss.id] = boss`
- 多处逻辑以 `bossId` 作为世界状态、击杀状态、刷新样本的主键

这对世界 Boss 没问题，但对战争使者不够。

因此后续实施时，至少要完成这类结构性调整：
- 在“目标定义层”区分 `single_npc` 与 `location_rule`
- 新增“按观测上下文匹配目标”的能力
- 不再假定所有目标都可以单纯依赖 `bossId` 唯一定位

注意：
- 这不要求一次性把所有存量数据结构完全推翻
- 但至少要让战争使者链路不再直接绑定到单 `bossId`

---

## 9. 数据兼容策略

## 9.1 目标

在不删除旧数据的前提下，引入新的显示与匹配能力。

## 9.2 原则

- 旧版世界 Boss 记录继续有效
- 旧版自定义目标记录继续有效
- 新增显示项配置时，尽量通过默认值兼容老存档
- 新增战争使者内置目标时，不要求旧存档中已有对应数据

## 9.3 建议方向

新增配置表时，采用默认值补齐方式，例如：
- 目标组显示配置
- 单项显示配置
- 战争使者内置目标启用状态

这样旧用户升级后：
- 能直接看到默认启用的目标
- 不会因为缺少新字段导致报错

---

## 10. 实施计划

以下为建议实施顺序。

### 阶段 1：目标定义层改造

目标：
- 让内置目标支持“单 ID 目标”和“地点型目标”两种模型。

工作项：
- 梳理现有默认目标定义结构。
- 为目标定义增加 `key`、`type`、`group` 等字段。
- 内置 5 个战争使者地点型目标。
- 保持现有世界 Boss 条目兼容。

完成标准：
- 代码层能表达战争使者“每张地图一个目标、每个目标允许多个 NPC ID”。

### 阶段 2：显示项配置

目标：
- 在设置页中引入“组开关 + 单项开关”。

工作项：
- 增加显示项配置的存档结构。
- 建立默认分组：世界 Boss、战争使者、自定义目标。
- 组开关关闭时，单项开关自动变灰。
- 补充说明文案：关闭显示会停止记录，但不删除已有历史。

完成标准：
- 用户可从设置页决定哪些目标显示与启用。

### 阶段 3：主窗口过滤

目标：
- 主窗口与悬停窗口只显示当前启用的目标。

工作项：
- 调整目标列表生成逻辑，只返回“当前启用显示”的目标。
- 保持主窗口不分组。
- 保持现有上下两块布局。

完成标准：
- 设置页勾选变化后，主窗口显示项实时同步。

### 阶段 4：观测与击杀匹配逻辑改造

目标：
- 关闭的目标不再记录。
- 战争使者观测与击杀能正确归属到对应地图。

工作项：
- 为观测链路增加“按规则匹配目标”逻辑。
- 为击杀链路增加相同的目标归属逻辑。
- 战争使者通过 `allowedNpcIds + zone` 命中目标。
- 未启用目标直接跳过记录。

完成标准：
- 同一颜色 ID 出现在不同地图时，不再串记录。

### 阶段 5：数据展示与样本验证

目标：
- 下半区位面信息和刷新估计能正确反映战争使者地图级状态。

工作项：
- 校准下半区对战争使者的状态读取。
- 验证关闭 / 开启目标后的展示行为。
- 验证旧历史保留是否符合预期。
- 针对战争使者刷新规律继续积累样本，确认是否接近 30 分钟。

完成标准：
- UI 展示与底层数据归属一致。

### 阶段 6：测试补充

目标：
- 为新功能增加专项验证清单。

建议新增测试点：
- 战争使者 5 张地图分别记录不串线
- 同一 NPC ID 在不同地图被正确归属
- 组开关关闭后单项开关变灰
- 单项关闭后主窗口不显示且停止记录
- 重新开启后旧记录仍可显示
- 全部关闭后不再继续积累新数据

完成标准：
- 新增功能具备可重复验证路径

---

## 11. 暂不纳入本次实施的内容

以下内容本轮先不做：
- 把战争使者做成用户可自由编辑的复杂规则
- 在主窗口中增加分组标题
- 删除或迁移已有历史记录
- 一开始就引入更复杂的地图 + 子区域 + 多条件匹配体系
- 对战争使者刷新规律做超前假设并写死固定时间

---

## 12. 最终结论

本轮设计最终确定为：
- 战争使者改为插件内置硬编码目标
- 建模方式按“每张地图一个地点型目标”
- 每个战争使者目标允许匹配 3 个颜色 ID
- 设置页改为“组开关 + 单项开关”
- 组关闭时单项变灰
- 主窗口继续保持当前上下两部分结构，不在主界面分组
- 所有目标关闭显示即停止后续记录，但保留已有历史数据

这套方案兼顾了：
- 现有窗口结构的稳定性
- 战争使者监控准确性
- 设置页可控性
- 后续扩展其它特殊目标的可持续性

## 更新附录 A：主窗口交互方案更新

本附录用于覆盖前文中“主窗口上下双区 / 顶部内容标签页”的旧 UI 思路。目标模型、显示项配置、记录规则仍然有效；主窗口交互以下内容为最新推荐方案。

### A.1 新的核心判断

此前 UI 方案的主要问题不是单纯“信息太多”，而是多个维度都被放在同一级切换：
- 服务器视图
- 击杀 / 位面内容页
- 窗口尺寸随页面变化
- 列头、统计口径、内容结构随页面变化

结果是：
- 即使只有一个服务器，也会出现多个不同页面；
- 用户每切一次都会失去空间记忆与内容记忆；
- 战争使者、自定义目标变多后，页面跳变会更明显。

因此，新方向不再追求“击杀页 + 位面页”双页面结构，而是改为：
- 主界面只保留一张稳定的击杀总览表；
- 位面与刷新细节下沉为单目标详情 tooltip；
- 通过一个新的“行动摘要列”承载决策信息。

---

## 更新附录 B：最新推荐方案

### B.1 总体结构

主窗口最终推荐保留一页：
- 顶部只保留服务器切换、设置、关闭；
- 不再保留“击杀记录 / 位面信息”双主标签；
- 主体区域只显示击杀矩阵；
- 在目标名称右侧新增一列“行动摘要列”；
- 原位面页中的细节信息，统一收纳到行动摘要列的悬停 tooltip 中。

说明：
- 目标名称列保持稳定文本，不再承担主要详情交互；
- 完整的目标详情入口统一集中到行动摘要列；
- 主窗口负责“扫”，tooltip 负责“读”，摘要列负责“决策”。

### B.2 为什么这样更优

这套方案的收益：
- 页面数量从“服务器 × 内容页”压缩回“服务器视图”这一层；
- 主窗口骨架稳定，不再因切页而跳变；
- 用户扫描路径清晰：先看击杀矩阵，再看目标行动摘要；
- 位面信息从“整页低频扫描”转为“按目标读取的高质量详情”；
- 对战争使者这类目标数量增加的场景更友好。

---

## 更新附录 C：行动摘要列

### C.1 定位

行动摘要列不是简单的“下一只刷新时间”文本列，而是：
- 当前目标的状态摘要；
- 用户是否值得立即行动的判断入口；
- 某个目标详情 tooltip 的唯一主交互热点。

它应同时承担：
- 读数
- 强提醒
- 可交互提示

### C.2 交互规则

行动摘要列：
- 支持鼠标悬停；
- 悬停后弹出该目标的详情 tooltip；
- 视觉上应明显区别于普通击杀格子；
- 必须让用户一眼知道“这里可以看更多信息”。

确认结论：
- 目标完整悬停详情不再放在目标名称列；
- 红框位置保留稳定文本标签；
- 蓝框位置成为交互主热点。

### C.3 视觉强化原则

行动摘要列建议使用以下手段强化可发现性：
- 高亮背景色或描边；
- 与普通击杀格不同的前景色；
- 不同状态使用差异化颜色；
- 明确的 hover 态；
- 仅在高价值状态上使用轻量动画。

动画原则：
- 用来“提醒”，不是用来“装饰”；
- 只对高价值状态使用；
- 优先使用低频、轻幅度的呼吸感或边框辉光；
- 禁止强烈、持续、高频闪烁。

不建议：
- 全列统一不停闪动；
- 所有摘要对象同等强度高亮；
- 用动画替代清晰文案。

### C.4 摘要列的语义目标

用户扫这列时，应快速知道：
- 这个目标是“快刷新了”还是“还没到”；
- 是“已有可靠预测”还是“样本不足”；
- 当前只有观测，还是已有击杀计时；
- 是否值得立刻切角色 / 切地图 / 去蹲守。

## 更新附录 D：行动摘要列状态建议

行动摘要列不建议只显示单一时间点，而应显示“状态摘要”。

建议状态包括：
- `-`
  含义：暂无数据。
- `仅观测`
  含义：有位面观测，但尚无可用击杀计时。
- `样本少`
  含义：已有计时，但样本不足，预测仅供参考。
- `8m 后`
  含义：预计还需约 8 分钟进入刷新窗口。
- `12:34`
  含义：已有相对稳定的预测刷新时刻。
- `窗口中`
  含义：当前已进入预测刷新窗口，应优先关注。
- `已过`
  含义：预测窗口已过，但尚未出现新的确认记录。

颜色语义建议：
- 灰色：无数据 / 占位
- 蓝色：仅观测
- 黄色 / 橙色：样本不足、参考状态
- 绿色：窗口接近或已进入
- 红橙色：已过或需要人工确认的异常状态

说明：
- 颜色不是唯一表达方式；
- 文案本身必须足够清晰；
- 即使弱化颜色，也应能独立读懂状态。

## 更新附录 E：Tooltip 重点规划

### E.1 Tooltip 的定位

Tooltip 不是“把旧位面页原文塞进去”，而是：
- 某个目标的详情卡片；
- 用户在主表中理解该目标状态的入口；
- 位面、观测、击杀、样本、预测信息的结构化阅读载体。

### E.2 Tooltip 的信息分层

推荐分三层：

#### 第一层：当前最重要的信息

优先回答：
- 当前是否已有活跃击杀计时；
- 当前最接近的预测刷新结果；
- 当前是否已进入刷新窗口；
- 当前仅有观测还是已有击杀样本。

这是 tooltip 的抬头区，也是用户最先看的部分。

#### 第二层：位面与最近活动详情

建议显示：
- 最近活跃位面；
- 最近观测时间；
- 最近观测角色；
- 最近击杀时间；
- 最近击杀角色；
- 当前相关地图 / 区域信息。

这部分承接原下半区的核心价值，但按“单目标详情”重组。

#### 第三层：样本与预测质量

建议显示：
- 样本数；
- 固定刷新 / 区间刷新；
- 预测窗口范围；
- 置信度；
- 最近实测刷新间隔。

这部分适合高级用户判断预测是否可信。

### E.3 Tooltip 的文案顺序建议

推荐顺序：
1. 目标名称
2. 当前状态摘要
3. 最近预测结果
4. 最近观测 / 击杀摘要
5. 位面详情
6. 样本与置信度

原则：
- 先给结论，再给证据；
- 先回答“现在要不要去”，再回答“为什么这么判断”。

### E.4 Tooltip 的长度控制

Tooltip 不能无限增长。

控制原则：
- 不堆叠重复信息；
- 避免把所有位面逐项平铺；
- 优先展示“最新 / 最相关 / 最有行动价值”的内容；
- 长期样本列表如需保留，应另设调试入口，不适合全部塞进 tooltip。

## 更新附录 F：导航层级更新

新的导航原则：
- 顶部只保留服务器切换；
- 不再让“服务器”和“内容类别”同时处于一级导航；
- 主窗口只有一个内容页：击杀总览；
- 详情由行动摘要列悬停承载。

这样可以避免：
- 1 个服务器也出现 6 个主页面；
- 页面切换导致窗口结构整体跳变；
- 用户频繁在“先切服务器还是先切内容”之间做决策。

## 更新附录 G：当前最新结论

截至本次更新，主窗口交互的最新推荐方案为：
- 主界面只保留击杀总览页；
- 顶部只保留服务器切换、设置、关闭；
- 原位面页取消作为主页面存在；
- 在目标名右侧新增“行动摘要列”；
- 目标完整详情统一收口到行动摘要列 tooltip；
- 摘要列通过高亮、颜色、hover 态、轻量动画强化其可交互性；
- 目标名称列保持稳定，不承担主要详情交互；
- 主窗口负责“扫”，tooltip 负责“读”，摘要列负责“决策”。

## 更新附录 H：准实施细化定义

本附录用于把“行动摘要列 + 详情 tooltip”细化到可以直接落代码的程度。

### H.1 主表列结构

主表最终列结构建议为：
- `Boss / 目标`
- `行动`
- 角色击杀列若干

说明：
- `Boss / 目标` 只负责稳定显示名称；
- `行动` 是唯一的详情悬停入口；
- 角色列继续只表达击杀状态，不混入位面细节。

### H.2 行动摘要列标题

推荐标题：
- `行动`

不推荐：
- `刷新`
- `预测`
- `位面`

原因：
- 这一列不只是显示时间；
- 它表达的是“现在该不该动、为什么该动”；
- 用 `行动` 更符合它的摘要与决策定位。

### H.3 摘要状态优先级

若一个目标在当前视图下存在多条可用状态，建议按以下优先级选出主状态：

1. `窗口中`
2. `Xm 后`
3. `HH:MM`
4. `已过`
5. `样本少`
6. `仅观测`
7. `-`

解释：
- 先强调“立刻值得去”的目标；
- 再强调“快到了”的目标；
- 之后再显示参考级信息；
- 最后才是弱信息和空状态。

### H.4 各状态的建议文案

推荐文案：
- `-`
  含义：当前无有效记录。
- `仅观测`
  含义：有位面观测，但没有可用击杀计时。
- `样本少`
  含义：有击杀与预测基础，但样本不足，不宜强信。
- `8m 后`
  含义：预计约 8 分钟后进入刷新窗口。
- `12:34`
  含义：存在相对稳定的目标刷新时刻。
- `窗口中`
  含义：当前已进入预测刷新窗口，优先关注。
- `已过`
  含义：预测窗口已过，等待新的确认信息。

显示规则建议：
- 若距离预测窗口开始时间小于 10 分钟：优先显示 `Xm 后`
- 若距离更远：显示 `HH:MM`
- 若当前处于预测窗口内：显示 `窗口中`
- 若预测窗口结束后仍无新信息：显示 `已过`

### H.5 摘要列颜色与强调

推荐颜色语义：
- `-`：灰色
- `仅观测`：蓝色
- `样本少`：橙黄色
- `Xm 后`：绿色
- `HH:MM`：青绿色或亮黄绿色
- `窗口中`：高亮绿色
- `已过`：橙红色

强调规则：
- `窗口中`：最强强调
- `Xm 后`：次强强调
- `样本少`：中等强调
- 其它状态：普通强调

推荐视觉层次：
- 颜色
- 边框亮度
- hover 亮起
- 仅对 `窗口中` 与 `Xm 后` 使用轻量动画

### H.6 动画范围

建议只对以下状态使用轻量动画：
- `窗口中`
- `Xm 后` 且 `X <= 10`

动画形式建议：
- 低频呼吸光效
- 轻微边框辉光变化

不建议：
- 文本持续闪烁
- 高频跳色
- 大面积背景脉冲

### H.7 Tooltip 字段顺序

Tooltip 推荐固定顺序：

1. 目标名称
2. 当前状态摘要
3. 最近预测结果
4. 最近位面 / 区域
5. 最近观测时间
6. 最近观测角色
7. 最近击杀时间
8. 最近击杀角色
9. 样本数
10. 预测模式（固定 / 区间）
11. 预测窗口
12. 置信度
13. 最近实测刷新

### H.8 Tooltip 字段显示规则

不是所有字段都必须显示。

建议规则：
- 无值则不显示；
- 同类字段避免重复；
- “最近位面 / 区域”只显示最相关的一条；
- 若只有观测无击杀，则隐藏击杀相关字段；
- 若无预测样本，则隐藏预测窗口与置信度；
- 若无最近实测刷新，则不显示该项。

### H.9 Tooltip 的首屏目标

用户悬停后第一眼应看到：
- 这个目标现在是否值得去；
- 最近的预测时间或窗口；
- 当前是哪一类信息支持这个判断。

因此 tooltip 首屏应尽量压缩为：
- 1 行名称
- 1 行状态结论
- 1-2 行关键依据

其余字段放在后续内容中。
